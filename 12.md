# 第十二课 为什么三个苹果和两个面包也只算一种选择？

## 第一部分 五份补给到底有多少种装法？

皇家保险箱终于换上了新密码。

这一次，红石工程师没有把密码写在箱子背面，也没有使用国王生日，更没有继续采用那组被鸡踩出来的数字。为了防止密码再次被偶然碰中，他还给四个转轮加装了透明活板门。

鸡站在活板门外看了很久。

它似乎第一次意识到，王国的技术进步有时会专门针对自己。

保险箱的问题解决以后，北方远征队又遇到了新的后勤问题。

村民食堂准备为每名队员发放五份食物。可供选择的食物一共有三种：

苹果。

面包。

胡萝卜。

同一种食物可以领取很多份。队员可以拿五只苹果，也可以拿三块面包和两根胡萝卜，还可以把三种食物混在一起。

食堂管理员将一只空补给箱推到托马斯面前。

“每箱必须正好装五份食物。”他说，“我想知道一共有多少种不同搭配。”

托马斯看着桌上的苹果、面包和胡萝卜。

上一课的密码有四个位置，每个位置可以从十个数字中任选一个，数字还允许重复，所以密码数量是：

\[
10^4
\]

现在有五份食物，每一份都可以选择苹果、面包或胡萝卜。按照同样的想法，似乎应该是：

\[
3^5=243
\]

“二百四十三种。”托马斯说道。

食堂管理员没有反驳，只按照托马斯的方式装出第一箱：

苹果、苹果、苹果、面包、胡萝卜。

接着，他换了一个放入顺序：

胡萝卜、苹果、面包、苹果、苹果。

“这是第二种吗？”管理员问。

托马斯看了看两只补给箱。

第一箱有三只苹果、一块面包和一根胡萝卜。

第二箱也是三只苹果、一块面包和一根胡萝卜。

食物进入箱子以后，不会保留领取顺序。队员也不会因为管理员先放胡萝卜，便认为自己的午餐发生了本质变化。

“不是。”托马斯说道，“这两箱搭配相同。”

管理员又装出：

面包、苹果、苹果、胡萝卜、苹果。

“第三种？”

“仍然是同一种。”

五份食物只要排列顺序不同，\(3^5\) 就会把它们当成不同结果。可国王关心的不是食物按照什么顺序进入箱子，而是最终每种食物分别有多少份。

三个苹果、一个面包、一个胡萝卜，无论怎样排列，都只能算一种补给搭配。

“那我用普通组合。”托马斯说道。

三种食物中选择五份：

\[
\binom35
\]

他刚写完便停住了。

从三个不同对象中选择五个，而且不允许重复，根本不可能完成。普通组合默认每个对象最多选择一次，可食堂显然拥有不止一只苹果，也不止一块面包。

这里允许重复选择同一种食物。

但选择的顺序又不重要。

它既不是上一课的密码，也不是普通的五人巡逻队。

食堂管理员又摆出几只补给箱：

五只苹果。

四只苹果和一块面包。

四只苹果和一根胡萝卜。

三只苹果和两块面包。

三只苹果、一块面包和一根胡萝卜。

托马斯决定按照苹果数量分类。

如果拿五只苹果，其他两种食物都不能再拿。

如果拿四只苹果，剩下的一份可以是面包或胡萝卜。

如果拿三只苹果，剩余两份可以是两块面包、一块面包加一根胡萝卜，或者两根胡萝卜。

继续这样列当然能够完成，但每次苹果数量减少，面包与胡萝卜之间又会产生新的分配方式。食物种类一旦增加，表格也会迅速变宽。

他需要找到一种更统一的表示方法。

那只被挡在保险箱外的鸡此时已经来到了食堂。它跳上桌子，叼走一颗用来记录数量的木质圆片，又把圆片丢进胡萝卜筐里。

食堂管理员捡回圆片。

“鸡算不算一种食物？”

托马斯看了它一眼。

鸡也看着托马斯。

这是一个在Minecraft里不适合当着鸡面讨论的问题。

“不算。”托马斯说道，“我们现在只统计苹果、面包和胡萝卜。”

鸡发出一声不满的叫声，转身跳进了空补给箱。它似乎认为，既然不能成为食物，至少应该成为补给内容。

托马斯把它抱出来，在桌面上放下五枚完全相同的木质圆片。每一枚圆片代表一份食物。

他没有立即给圆片写上苹果、面包或胡萝卜，而是重新问起那个越来越熟悉的问题：

**一箱食物，究竟由什么信息唯一决定？**

不是拿取顺序。

不是五份食物分别占据哪个位置。

真正重要的只有三个数量：

苹果有几份。

面包有几份。

胡萝卜有几份。

三个数量加起来必须等于五。

如果用 \(a\)、\(b\)、\(c\) 分别表示三种食物的数量，那么托马斯需要统计的，其实是：

\[
a+b+c=5
\]

有多少组非负整数解。

“非负”意味着每种食物可以拿一份、两份，也可以一份都不拿，但不能拿负三根胡萝卜。即使财政大臣愿意把负数写进账本，食堂也无法把三根不存在的胡萝卜装进箱子。

托马斯盯着五枚圆片，忽然想起仓库里用来分隔货物区域的木栅栏。

如果能够用两道隔板，把五份相同的食物分成三段，那么每一段的圆片数量，不就恰好能够表示一种食物的份数吗？

他找来两根细木条，放到五枚圆片之间。

一个新的数法，开始出现了。

## 第二部分 五颗星与两块隔板

托马斯先把五枚圆片排成一行：

```text
● ● ● ● ●
```

它们暂时不代表具体食物，只代表一共需要分配的五份补给。

接着，他加入两块隔板。

第一块隔板左边的圆片，代表苹果。

两块隔板之间的圆片，代表面包。

第二块隔板右边的圆片，代表胡萝卜。

例如：

```text
● ● ● | ● ● |
```

表示：

苹果三份。

面包两份。

胡萝卜零份。

也就是三只苹果和两块面包。

如果隔板这样摆：

```text
● | ● ● | ● ●
```

则表示：

一只苹果、两块面包和两根胡萝卜。

如果第一块隔板放在最前面：

```text
| ● ● | ● ● ●
```

表示没有苹果，两块面包和三根胡萝卜。

如果两块隔板紧挨在一起：

```text
● ● ● ● ● | |
```

表示五只苹果，面包和胡萝卜都是零份。

相邻隔板不是错误。

它代表夹在两块隔板之间的那种食物数量为零。

隔板位于最前面或最后面也完全合法，因为某一种食物可以一份都不选。

托马斯不断移动两块隔板，很快发现了一件漂亮的事情。

每一种食物搭配，都能写成一种唯一的“圆片与隔板”排列。

三只苹果、两块面包、零根胡萝卜，只能写成：

```text
● ● ● | ● ● |
```

不会有第二种写法。

反过来，每一种由五枚圆片和两块隔板组成的排列，也都会对应唯一的食物搭配。

只要数清这些排列，就等于数清所有补给箱。

这又是一种一一对应。

托马斯不再直接数食物，而是数：

五枚相同圆片和两块相同隔板，能够形成多少种不同排列。

总共一共有七个位置：

五个位置放圆片。

两个位置放隔板。

只要从七个位置中选出两个，用来放隔板，剩下的位置自然全部放圆片。

因此，方案数是：

\[
\binom72=21
\]

也可以选择五个位置放圆片：

\[
\binom75=21
\]

两种写法相等，因为选择哪两个位置放隔板，也同时决定了哪五个位置没有放隔板。

食堂管理员开始按照隔板排列检查补给箱，最终确实得到二十一种搭配。

五份全是苹果。

五份全是面包。

五份全是胡萝卜。

四份一种食物，加一份另一种。

三份、两份。

三份、一份、一份。

以及各种其他分配。

没有哪一种搭配被重复，也没有任何搭配遗漏。

后来，数学家把这种方法称为**隔板法**，英文里也常叫 **Stars and Bars**，也就是“星星与隔板”。

圆片通常被画成星星：

```text
★ ★ ★ | ★ ★ |
```

星星代表需要分配的相同物品。

隔板负责把这些物品分到不同类别。

名字听起来像一家同时出售天文仪器和建筑材料的商店，实际却只是一个非常聪明的计数转换。

一般来说，如果从 \(n\) 种物品中选择 \(m\) 份：

同一种物品可以重复选择。

只关心每种物品最终选择了多少份。

不关心选择顺序。

那么可以使用：

\(m\) 颗星星，表示需要选择的 \(m\) 份物品。

\(n-1\) 块隔板，把星星分成 \(n\) 段。

一共需要排列：

\[
m+n-1
\]

个符号。

只要选择其中 \(n-1\) 个位置放隔板，便得到：

\[
\binom{m+n-1}{n-1}
\]

种方案。

根据组合数的对称性，也可以写成：

\[
\binom{m+n-1}{m}
\]

所以，三种食物中选择五份的数量是：

\[
\binom{5+3-1}{3-1}
=
\binom72
=
21
\]

这种计数模型通常被称为**可重复组合**。

托马斯把它与前几课的方法放在一起比较。

从 \(n\) 个不同对象中选择 \(m\) 个，不允许重复，顺序不重要：

\[
\binom nm
\]

从 \(n\) 种对象中选择 \(m\) 次，允许重复，而且顺序重要：

\[
n^m
\]

从 \(n\) 种对象中选择 \(m\) 份，允许重复，但顺序不重要：

\[
\binom{n+m-1}{m}
\]

三种问题看起来只改了几个字，答案却完全不同。

普通组合中，一个对象一旦选中，不能再选。

密码中，同一个符号可以不断出现，而且每个位置有区别。

重复组合中，同一种类型可以反复选择，但同样数量的最终搭配只算一次。

铁匠拿来几种情况检验托马斯。

“红、黄、蓝三种羊毛，铺一条五格长的装饰带。”

“每格位置不同，颜色允许重复，顺序重要，所以是：

\[
3^5
\]
”

“三种羊毛一共领取五块，全部放进同一只箱子，只关心各色数量。”

“重复允许，顺序不重要，所以是：

\[
\binom72
\]
”

“三种不同颜色的羊毛中选择两种，每种最多选一次。”

“普通组合：

\[
\binom32
\]
”

“给三名骑士分发五块完全相同的面包，每块面包必须发出去，每名骑士可以拿零块。”

托马斯在桌面上摆出五颗星星和两块隔板。

“仍然是把五个相同对象分成三组，所以也是：

\[
\binom72
\]
”

物品变成食物、羊毛或面包，并不会改变计数结构。

只要它们满足：

总数固定。

类别数量固定。

同类物品不区分身份。

每类可以得到零个或多个。

隔板法就可能出现。

国王看着二十一种补给搭配，又提出了一个新要求。

“每只补给箱里，三种食物都必须至少有一份。”

这一次，隔板不能放在最前面或最后面，两块隔板也不能紧挨在一起。因为这些情况都会让某一类食物数量变成零。

托马斯没有重新研究所有受限制的隔板位置。

他先给每种食物各放一份。

一只苹果。

一块面包。

一根胡萝卜。

五份补给已经使用三份，还剩两份可以自由分配给三种食物。

于是，问题变成：

从三种食物中再选择两份，允许重复，顺序不重要。

方案数是：

\[
\binom{2+3-1}{3-1}
=
\binom42
=
6
\]

“为什么先给每种食物一份？”国王问。

“因为‘至少一份’是必须完成的部分。”托马斯回答，“先把它满足掉，剩余数量就重新变成可以为零的普通重复组合。”

一般来说，将 \(m\) 个相同物品分给 \(n\) 个类别，并要求每类至少一个，必须先保证：

\[
m\ge n
\]

随后给每类一个，剩下：

\[
m-n
\]

个自由分配。

方案数便是：

\[
\binom{(m-n)+n-1}{n-1}
=
\binom{m-1}{n-1}
\]

托马斯没有急着把这个新公式单独背下来。

他只记住了更自然的方法：

**遇到“每类至少一个”，先给每类一个，再处理剩余部分。**

公式只是这个动作的缩写。

Notch直到所有补给箱摆满桌面时才走进食堂。

他看了看五枚圆片和两块隔板，没有询问答案。

“你这次为什么没有数五个位置？”他问。

“因为食物进入箱子后，位置没有意义。只要每种食物的数量相同，就是同一种搭配。”

“为什么隔板有两块？”

“三种食物需要切成三段，只需要两道分界。”

“如果有十种食物呢？”

“需要九块隔板。”

Notch拿起其中一块隔板。

“所以你没有直接排列食物。”

“我排列的是能够唯一表示每种数量的星星与隔板。”

Notch点了点头。

托马斯已经能够主动更换计数对象。

食物搭配不好数，就数隔板排列。

长方形不好数，就数边界。

连续区间不好数，就数端点。

很多看似不同的计数技巧，其实都来自同一种能力：

**不要执着于直接数眼前的东西。**

**寻找另一种更容易数、又与答案一一对应的表示。**

## 第三部分 程序员时间：机器只记录三种食物各有多少

红石工程师准备让机器列出全部二十一种补给搭配。

他最初仍按照密码程序的方式，设置五个食物位置，每个位置可以选择苹果、面包或胡萝卜。

机器很快开始输出：

```text
苹果 苹果 苹果 面包 胡萝卜
```

随后又输出：

```text
胡萝卜 苹果 面包 苹果 苹果
```

接着是：

```text
面包 苹果 苹果 胡萝卜 苹果
```

托马斯看了前三行。

“这三行是同一种搭配。”

工程师点点头。“顺序不同。”

“但我们不关心顺序。”

如果先生成 \(3^5=243\) 个有顺序序列，再把数量相同的结果合并，当然能够得到二十一种。可这意味着机器先制造大量重复，再努力删除自己刚刚制造的重复。

人类有时也会采用这种方法，并把后半部分称为“数据清洗”。

托马斯让程序直接记录每种食物的数量。

苹果可以从零份到五份。

苹果数量确定以后，面包可以从零份到剩余数量。

最后的胡萝卜数量无需继续选择，因为总数必须是五。苹果和面包确定后，胡萝卜自然是剩下的部分。

```cpp
#include <iostream>
using namespace std;

long long choose(int n, int k) {
    long long result = 1;

    for (int i = 1; i <= k; i++) {
        result = result * (n - i + 1) / i;
    }

    return result;
}

int main() {
    int total = 0;

    for (int apple = 0; apple <= 5; apple++) {
        for (int bread = 0;
             bread <= 5 - apple; bread++) {
            int carrot = 5 - apple - bread;

            cout << apple << " 苹果，"
                 << bread << " 面包，"
                 << carrot << " 胡萝卜\n";

            total++;
        }
    }

    cout << total << '\n';
    cout << choose(7, 2) << '\n';
}
```

程序最后输出：

```text
21
21
```

第一个二十一来自逐种枚举食物数量。

第二个二十一来自隔板公式：

\[
\binom72
\]

两种方法再次得到相同答案。

程序没有列出五份食物的每一种领取顺序。它只列出苹果、面包和胡萝卜的最终数量，因此每种补给搭配只出现一次。

“为什么不需要第三层循环枚举胡萝卜？”托马斯问。

工程师回答：“因为总数固定为五。苹果和面包确定以后，胡萝卜已经没有选择。”

这与边界计数十分相似。

有些信息并不需要全部独立选择。

只要前面的信息确定，最后一个量会被总数自动决定。

Notch站在打印机旁，拿起两张记录：

```text
3 苹果，2 面包，0 胡萝卜
```

以及：

```text
3 苹果，2 面包，0 胡萝卜
```

“如果两张纸完全一样呢？”他问。

“就是同一种搭配，程序只应该打印一次。”

“机器为什么没有重复？”

“因为它不再枚举食物顺序，只枚举最终数量。”

Notch放下纸条。

程序能否避免重复，很大程度上取决于人选择用什么方式表示一个答案。

使用五个有顺序位置，重复会大量出现。

使用三个食物数量，每种搭配天然只有一种写法。

最好的去重，有时不是生成以后再删除。

而是一开始就不要制造重复。

当天傍晚，国王把托马斯叫到议事厅。

长桌上摆着过去所有任务的记录：

皇家仓库的木材统计。

骑士装备的完整搭配。

规则排列的新仓库。

农田里的长方形。

仪仗队站位。

五人巡逻队。

远征背包。

铁路车票。

重叠的庆典名单。

至少携带一种补给。

皇家保险箱密码。

以及五份食物的二十一种搭配。

国王没有再问其中任何一道题的答案。

他只问道：

“如果明天有人带来一个新的问题，你准备先做什么？”

托马斯没有翻公式，也没有立刻寻找可以相乘的数字。

他看着这些曾经让自己数错、重数、漏数，甚至被苦力怕炸掉过的记录。

然后回答：

“我会先问，这一次到底在数什么。”

国王继续问：“然后呢？”

托马斯拿起一张空白纸。

有些问题需要判断顺序是否重要。

有些需要判断能否重复选择。

有些对象必须连续，有些可以任意跳过。

有些类别互不重叠，有些会把同一个人记录很多次。

有时直接数目标容易，有时数反面更短。

有时真正应该数的不是答案本身，而是边界、隔板，或者另一种与答案一一对应的东西。

这些问题终于不再是一堆互不相关的技巧。

它们正在组成一张完整的地图。

下一课，托马斯不再需要发现一种新的公式。

他需要做的是把已经走过的道路整理起来，回答那个比所有答案都更重要的问题：

**面对一个全新的计数问题，第一步究竟应该往哪里走？**
